## Dependency notes

- Plans 001, 002, and 003 have no direct dependencies and can be executed in
  separate worktrees. They touch different source areas except for shared
  package-level verification.
- If executing in one branch instead of isolated worktrees, run them in numeric
  order to keep review smaller.
- Plans 004 and 005 (added 2026-06-18) are the "make Express/multer optional via
  DI" pair, and are **independent of each other** and of 001–003. Both are
  **Phase-1 only**: they introduce the seam (a peer dep for multer; an
  `ExpressService` for Express) without yet removing the hard deps. The real
  install-time optionality (dropping `express`/`cors`/`multer` from
  `@agentback/rest`'s `dependencies`) is **Phase 2**, gated on flipping the
  default listener to `'native'` — the `fetch-seam-root-cutover` spec's
  "item D / full demotion". Do not attempt Phase 2 inside 004/005.
- **Key caveat carried in both plans:** DI / peer-deps fixes the npm *install*
  graph, not the esbuild *bundle* graph. The dep must live in a module the edge
  build never statically imports (subpath imports, not the `@agentback/express`
  barrel), or it re-enters a Worker bundle. The bundle doctor on
  `packages/cli/fixtures/cf-app` (`{ok:true}`) is the gate for this in both.
- 004 is the lower-risk near-term win (the Web multipart adapter already exists);
  005 is core-`RestServer` surgery — land 004 first if sequencing.
- Plan 006 is the **Phase-2 design spec** (added 2026-06-18, revised to the
  two-class design): why "drop express from rest's deps" is a no-op (express is
  transitive via `rest → @agentback/express → express`), and the path —
  `EdgeRestApplication` (neutral, no express) + `ExpressRestApplication` (Express
  host) over a neutral-package split. P2.0 was measured (45 fails under a global
  native default), but the two-class design **reframes those gaps as the edge
  class's incremental feature backlog, not a gate** — so the core (P2.1 extract →
  P2.2 the classes → P2.3 drop express from the edge install) is a contained
  increment, no longer blocked on the item-D global default flip. P2.4 (porting
  every Express feature to the fetch path) stays open-ended / demand-driven.

## Findings considered and rejected

- SSRF in `mcp-connect`: rejected. `RemoteRegistry` defaults to blocking
  loopback/private/reserved targets and wraps OAuth discovery/token fetches with
  guarded fetches. Existing tests cover default-deny behavior.
- Scaffold path traversal in `create-agentback`: rejected. App names are
  validated as npm package names, scoped packages use only the part after `/` as
  the directory, and non-empty targets are refused.
- Plugin DI key collisions: rejected. `loadPlugins` snapshots binding
  identities and fails closed on collisions unless a key is explicitly listed in
  `allowOverride`.
- Release tag/version reproducibility: valid but lower priority than the three
  plans above. Revisit after the security/test coverage plans land.
- OpenAPI/AX document caching: valid but lower priority until route/spec size or
  request volume makes the repeated assembly measurable.

Actors are addressable state

Some state refuses to be a stateless handler over a database row. A shopping cart, a chat session, a game room, a rate counter — each one needs a single writer at a time, picked by identity. The usual answers are a pile of row locks or an external actor runtime that wants to own your process model. AgentBack's actor runtime takes a smaller step: an actor is the framework you already have, plus one guarantee.

Reach for the naive version and the bug writes itself. Two requests load the same cart, each adds an item to its own copy, each saves — and one write clobbers the other. The fix is to make access to a given cart serial: one turn at a time, for that cart, while other carts run freely. You can buy that with per-row locking and a lot of care, or you can adopt an actor framework that asks you to write your domain logic inside its abstractions and its lifecycle. AgentBack's actors offer that serialization without the second runtime.

An actor is a DI service with a stable address

The model is deliberately small: an actor is a DI service implementing a typed state machine with a stable address. The key discipline is that state is an explicit argument and return value, never an instance field. The service object holds behavior and injected dependencies; the state lives in the runtime. That one rule is what makes turns serializable, failures rollback-able, and the backing store swappable. 

@actor('cart', {state: CartState})
class CartActor implements Actor<z.infer<typeof CartState>> {
  constructor(@inject('services.Catalog') private catalog: Catalog) {}

  initialState() {
    return {items: {}};
  }

  @actorCommand('add', {input: AddItem, output: CartView})
  add(state: z.infer<typeof CartState>, input: z.infer<typeof AddItem>) {
    this.catalog.assertExists(input.sku); // AgentError -> 400 on unknown SKU
    state.items[input.sku] = (state.items[input.sku] ?? 0) + input.qty;
    return {state, result: cartView(state)};
  }
}

The same Zod schemas that validate the command and the result are the z.infer TypeScript types and the contract the REST route publishes. The actor is a normal service, so it injects a Catalog through the same container as everything else, and a domain error it throws (an AgentError) reaches the caller as a 400 instead of a redacted 500. Nothing here is a new execution model — it's a decorator and a method.

The address is the API

cart/<id> is the unit of both state and serialization. Two ids are independent; the same id is one logical line of turns. A REST controller is simply a caller — actors do not become endpoints by themselves, which keeps the transport decision yours: 

@post('/{id}/items', {path: CartPath, body: AddItem, headers: IdempotencyHeaders, response: CartView})
async add(input) {
  const turn = await this.carts.invoke(
    'cart',
    input.path.id,                              // the actor's address
    {name: 'add', input: input.body},
    {requestId: input.headers['idempotency-key']}, // the turn's identity
  );
  return turn.output;
}

The cart id in the URL is the address. The Idempotency-Key header becomes the turn's requestId, and that is where the state discipline pays off: because state is a value the runtime clones before each turn, a handler that throws or returns a schema-invalid result commits nothing, and a retried request with a committed requestId replays the stored result instead of running add twice. Reusing that key for a different payload is rejected. A double-clicked "add to cart" is a no-op, for free.

The runtime is a port, not a framework

The decorated class is the authoring model; the runtime contract underneath it — ActorRuntime, with a normalized ActorDefinition — has no idea decorators or DI exist. That boundary is the whole point. The in-memory adapter is a single-process reference for tests and API validation. Swap it for the Redis adapter and the same actor serializes across processes: a per-identity lease coordinates one turn at a time, and the next state plus the dedup result commit atomically in a Lua script. The actor and the controller do not change a line: 

// instead of: this.component(InMemoryActorsComponent)
installRedisActors(this, {connection: {url: process.env.REDIS_URL}});

Every adapter has to pass the same conformance suite — per-identity serialization, cross-identity concurrency, rollback, request replay, collision rejection, and an atomic commit of state plus result. A queue acknowledgement next to an unrelated database write does not meet that contract, and the suite says so.

Reads and facts

The same boundary gives two more things almost for free. A @actorQuery reads the state without taking a turn — no lease, no mailbox slot — so reads run concurrently with writes and with each other; you ask the address a question instead of queuing behind it. And a command may return events alongside its result: domain facts the event-log runtime persists atomically with the state change and hands to subscribers. State is the boundary, a message is intent, an event is fact — and all of them cross the same address.

What it deliberately isn't

The honesty of this model is in its non-goals. There is no distributed directory or placement, no activation, passivation, reminders, or supervision, and no agent loop — AgentBack is not trying to become an orchestration runtime. The runtime can roll back actor state; it cannot un-send an email or un-charge a card, so durable actors still owe you an outbox or idempotent downstream calls. The next real acceptance gate is one adapter on a backend with native per-key serialization and transactional storage — Cloudflare Durable Objects — proving crash and retry behavior, not just the in-process contract.

A cart is not a row, and an actor is not a runtime. It's an address, a schema, and one turn at a time — the pieces AgentBack already had, pointed at the one problem stateless handlers can't solve.
